Medium-scale abstraction: Choose a part of the project that you (co-)wrote that employs medium-scale abstraction.
--------------------------------------
WHERE: scenes/manager.lua, all scene files in scene/..
MY CONTRIBUTION: Coming up with the conceptual structure of the scene transitions, writing the code for manager.lua, creating the folder and sub-folder structure, as well as the
scene transition calls across all the scenes. I used online reference to love2d and lua in coming up with this format, and used the help of genAI copilot in helping me with the code format and debaugging. 

DESCRIPTION:
- What functionality does your code sample abstract away?
The process of changing between menu items in the sub-menus of the game, so that no files need to be called explicitly. Instead of directly loading a file of the sub-menu, and creating a format to account for user interaction
-- such as mouse clicks or pressing keys, but also accounting for love2d core operations -- updating, drawing and loading. This way I can have a structured control over all transitions abstracted to the form of "scenes" instead of specific file paths.

- How does it mechanical achieve abstraction?
It is done by the assignment of a new "current" scene to the scene provided by a current "current" scene. The program starts with an initial scene of logo.lua, proceeds to the login scenes and to the main scene of the game, and then the user can transition 
between the scenes by interacting with the user interface. The role of manager.lua is to take in the directions from scenes on what scene comes next -- whether automatically (like logo--login transition) or on demand (button press in menu) and call for the 
scene information to be loaded. This includes retriving the file data through loadScene(), ensuring the existance of a file, and then facilitating all user-interaction funcitons (key press, mouse press, text input) and love2d core functionalities -- drawing,
updating and loading. 

- What purpose does this abstraction serve in your larger program?
It simplifies the code significantly. The abstraction takes what all the scenes have in common -- the need of handling user input and being responsive to love2d operations. It adds the conceptual structure to the way the scenes operate and provides a clean
functionality that requires a potential developer to only concern themselves with single-scene functionality but no need to be call each of the scene functionalities explicitly in the caller snippet.

Medium-scale architecture: Choose one of the components of your project that you (co-)designed according to its architecture, e.g., front-end versus back-end.
--------------------------------------
WHERE: stats_init.lua
MY CONTRIBUTION: Functions that interact with a statistics data file -- verify its existance, load it, check or change value and update loaded table. Specficially, functions: save_data(), stats_init.delete_save_file(), stats_init.change_stat(key, delta), 
is_game_existing(), stats_init.load_data(path), stats_init.data_setup(), stats_init.change_value(key, new_value), stats_init.reset_player_data(). I collaborated with Connor working on this file, but he was in charge of creating the statistical updates to the table
while I created the system of interacting with the text file that holds information between the login sessions.

- What architectural pattern does your program employ?
The program employs multilayer architectural pattern. There is a layer of interface presented to the user, the operations underlying their actions and managing the display of the interface, the operations interacting with the in-game data, as well as the operations 
that direct the interaction with data files. My specific contribution discussed here relates to the creation of the data file and the intractions with it that in turn inform back the in-game data and user interface.

- What components result from this pattern in your program?
The program divides the differnt layers into separate sets of directories with similar functionalities. For example, each of the scenes refers to the style sheet that imports the visual information, such as pop-up windows. These visual interface elements are directed
by the scenes, which are responsive to user interaction with the interface. Concurrently (or is it really concurrent, CSC213?) the program records user interactions and updates the in-game data, and then updates the data stored in out-of-game text file, to save the 
player's progress. The data and statistics manipulation is managed by the update() functionality of love2d that allows for backend changes, while draw() facilitates front-end presentation to the user. 

- What technologies and/or libraries make-up each of the components?
The game is created using love2d and lua language, and uses their features to employ our architecture. On of love2d's core function -- draw(), facilitates the display of visuals to the user; load() enables timed actions only upon the program startup, update() 
enables for regular updates of the backend information (statistics, saving to the files).
