WHERE: scenes/manager.lua, all scene files in scene/..
MY CONTRIBUTION: Coming up with the conceptual structure of the scene transitions, writing the code for manager.lua, creating the folder and sub-folder structure, as well as the
scene transition calls across all the scenes. I used online reference to love2d and lua in coming up with this format, and used the help of genAI copilot in helping me with the code format and debaugging. 

DESCRIPTION:
- What functionality does your code sample abstract away?
The process of changing between menu items in the sub-menus of the game, so that no files need to be called explicitly. Instead of directly loading a file of the sub-menu, and creating a format to account for user interaction
-- such as mouse clicks or pressing keys, but also accounting for love2d core operations -- updating, drawing and loading. This way I can have a structured control over all transitions abstracted to the form of "scenes" instead of specific file paths.

- How does it mechanical achieve abstraction?
It is done by the assignment of a new "current" scene to the scene provided by a current "current" scene. The program starts with an initial scene of logo.lua, proceeds to the login scenes and to the main scene of the game, and then the user can transition 
between the scenes by interacting with the user interface. The role of manager.lua is to take in the directions from scenes on what scene comes next -- whether automatically (like logo--login transition) or on demand (button press in menu) and call for the 
scene information to be loaded. This includes retriving the file data through loadScene(), ensuring the existance of a file, and then facilitating all user-interaction funcitons (key press, mouse press, text input) and love2d core functionalities -- drawing,
updating and loading. 

- What purpose does this abstraction serve in your larger program?
It simplifies the code significantly. The abstraction takes what all the scenes have in common -- the need of handling user input and being responsive to love2d operations. It adds the conceptual structure to the way the scenes operate and provides a clean
functionality that requires a potential developer to only concern themselves with single-scene functionality but no need to be call each of the scene functionalities explicitly in the caller snippet.

- What architectural pattern does your program employ?
The program employs multilayer architectural pattern. There is a layer of interface presented to the user, the operations underlying their actions and managing the display of the interface, the operations interacting with the in-game data, as well as the operations  that direct the interaction with data files. My specific contribution discussed here relates to the creation of the data file and the intractions with it that in turn inform back the in-game data and user interface.

- What components result from this pattern in your program?
The program divides the differnt layers into separate sets of directories with similar functionalities. For example, each of the scenes refers to the style sheet that imports the visual information, such as pop-up windows. These visual interface elements are directed by the scenes, which are responsive to user interaction with the interface. Concurrently (or is it really concurrent, CSC213?) the program records user interactions and updates the in-game data, and then updates the data stored in out-of-game text file, to save the  player's progress. The data and statistics manipulation is managed by the update() functionality of love2d that allows for backend changes, while draw() facilitates front-end presentation to the user. 

- What technologies and/or libraries make-up each of the components?
The game is created using love2d and lua language, and uses their features to employ our architecture. On of love2d's core function -- draw(), facilitates the display of visuals to the user; load() enables timed actions only upon the program startup, update()  enables for regular updates of the backend information (statistics, saving to the files).

-- delete_save_file
-- deletes the file, so that we can test whether the file gets created again
function delete_save_file(filename)
    if love.filesystem.getInfo(filename) then
        -- empty out the file for all but the headers
        local file = io.open(dir .. filename, "w")
        if not file then
            error("Failed to open file for writing: " .. dir .. filename)
        end
        file:write("id,first_name,last_name,age,sex,hometown,face_look_id,job,positive_relation,negative_relation\n")
        file:close()
    end
end

-- used later in the saving body of the code with:
newrun = true
if newrun then 
    delete_save_file("data_family.csv")
end

MY CONTRIBUTION: 
- The code snippet comes from my contribution to the project - a data file and a set of operations that underlie randomized generation of family members for the player. The code uses user's username to generate family members with the same last name but a different first name -- two parents and a randomized number of siblings. Their first names, and jobs are randomly selected from a data file available to the program, with a geneder specificity of the name to the parents' genders. I was in charge of creating that code, and also in charge of testing.

- According to code coverage tools, what aspects of this component are covered with tests?
The testing part presented here was used to delete the file when a "newrun" flag is selected. This was used as a testing measure to make sure that the program actually writes to the file at each time that the game is played, and that the name, age and jobs are actually random for each family member. This allowed also to verify whether a non-repetitive number of siblings is being selected correctly, and the formatting or resulting table saved into a data file is as expected. If not for the forceful flag-enforced deletion, these capacities would not be testable in regular game timeline - as we cannot afford to execute the entire game before checking these features.  

- What kinds of tests are they?
The test enforces the use of a a function that erases previously saved data of players' family members, so that the changes in recorded data file can be observed upon every iteration of game setup. To ensure the correct execution of the tested functionality I used this feature multiple times during the testing processes and recorded the data saved into the file to ensure the diversity of created family characters. Furthermore, the code throws an error if the file tunrs out to be missing, ensuring that the deletion function itself works correctly at wiping out only the necessary part of the data -- not deleting the file or erasing headings.

- For code that is not covered by tests, why did you not cover them?
A significant part of the code's functionality can be observed indirectly thanks to the use of this test -- for example, loading of the source data files (names, jobs), as the successful implementation of these is prerequisite to their diversed insertion into the data file. Also, while the code tests whether the information was randomly selected, it is not really an efficent testing process (which really is not a question in this sub-point, but is very relevant to my understanding and hopes for improvement of testing abilities), as it requires multiple games to be played and a level of assumption as to whether the random names are actually selected and we are not under illusion of random name/age/job selection. So in retrospect it would have been more accurate to create a testing suite that generates a list of say 100 names (same with jobs), and prints them into a testing file, wile calculating the number of times each appeared to ensure reasonable distribution.

    if not file then
    --error("Failed to open file for writing: " .. stats_init.save_filename)
        return false
    end

-- setup all data when called by main, and call save_to_main to indeed save to main
function stats_init.data_setup()
	-- if no game existing -- start new
	if not is_game_existing() then
        os.execute("mkdir -p " .. dir)
		stats_init.data = {}
		stats_init.create_new_player()
	-- if game existing -- load data from file to data table
	else
        local file = io.open(file_path, "r")
        local contents = file:read("*a") -- read the contents of the file
        file:close()

        if contents and contents ~= "" then
            stats_init.data = lume.deserialize(contents)
        else
            stats_init.data = {}
            stats_init.create_new_player()
        end

        if not is_game_existing() then
            error("Failed to create new player data file.")
        end
    end
end

How have you automated your tests so that they both take "1-click" to run and run automatically as part of build validation.
- The limited testing capacity of Lua made me think creatively of the testing strategies I can use to validate the correct execution of my code. Therefore, I used the flag "newrun" to mark whether the run was a test run or not. For example, in stats_back.loadCsvList() I was loading all the files with environment information (friends, classmates and family). These would normally occur only once, at the start of the game and then record the user interction with the game and save it for later retrieval. However, this limited my ability to check whether they are correctly done (whether they are randomized, and whether they behave correctly given different randomized data used from pre-created source files). So, using the flags as "1-click" testing option was very helpful in the testing process. However, it does not run concurrenly with the game exacution. On the other hand, there are features that do such as is is_game_existing(), which can assess whether the game data exist (meaning that it's not the first game run, and data needs retrieval). If an attempt to do so was made, but the data file was not successfully created, then an error would occur as shown towards the last lines of stats_init.data_setup(). This combines the practical development functionality of the testing with practical use for users needs.

Describe a specific occurrence in which your testing infrastructure saved you and/or your team work?
- It was really helpful to use the flag in debugging so that I can see a clear view of what the data saving situation is like. For example, when looking at the saved I realized that when recording parent information in setup_family.lua I would end up with columns misplaced for the parents, but not the children. That inclined me to investigate the issue and find out that the format of fetching jobs from a job source file sometimes resulted in an additional space being added to the job string, which translated to an additional column in the created cvs data file. Thanks to this testing feature I was able to experiment with different random formats and observe the changes in the formatting problems based on number of children created, or the types of jobs selected for the parents.

What techniques did you employ to discover users' needs?
I completed an observation of a person interacting with a game used as an inspiration for our project -- BitLife. For about 15 minutes I watched as they played with the application while recording some of their choices, patterns of behavior and presumed reactions to what was happening in the game.

What did you learn about users' needs from this process?
I found that the user was engaged in the game, and seemed to enjoy playing with it. It was interesting to see them gain an understanding of the game's functionality without explicit need for a tutorial - thanks to a very straight-forward user interface. They also seemed to be naturally drawn to positively impact the life of their character and set them up for a success in all dimensions of the game (so generally, improving the life scores, academic performance when in school and work performance when at workforce). That seemed to show that BitLife design can serve as a compelling model for our application

WHERE: Prototyping assignment submission
MY CONTRIBUTION: Conceptual and visual prototype of the game -- completed by myself

What design questions did your intend to answer with your prototype(s).
- I wanted to gain an understanding of how the design of BitLife can be adapted for our purposes -- what elements that we identified as poor design decisions in the BitLife model can be improved, and which ones are appealing to be inspired by in our design. Moreover, I wanted to sketch out the decision processes that go into using the game -- what kind of functionalites we may need to consider in implementation of the menus and buttons. For example, when do the users "return" to the main menu after completing an action -- is that right after performing it, or should they be allowed to manually switch back to the main menu, or remain in the sub-menu where their action was performed from instead?

How did you design your prototype(s) and subsequent user studies to address these questions?
- In my prototype I created a general sketch of the layout of the critical menu elements, so that it is clear what kind of design elements are presumed to be considered in our development. It served as a good starting point for the future developments of the visual design components, since we had the conceptual framework done. I also prepared decision trees - one general showing the visual aspects of the game scenes transition, and another one that treated the topic from rather conceptual perspective and showed a sample transition between sub-menu sections. 

What answers to your design questions did your efforts unveil?
- Most importantly, I realized that some of the BitLife functionalites were possibly too complex for us to employ given the timeline of the project -- they used a variety of design choices that would require more specificity in designing each sub-menu item than we intended to have. I also noticed how sketching out the prototype in resemblance to BitLife made it appear too "crowded" with the design elements, and suggested to me that it may be more appropriate to keep rather straight-forward design that does not take away from gameplay but is not as overstimulating (for example, the main menu includes a lot of information that otherwise could have been accessed by pressing onto a button for "more information")
