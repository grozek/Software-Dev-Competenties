    MY CONTRIBUTION: Coming up with the conceptual structure of the scene transitions, writing the code for manager.lua, creating the folder and sub-folder structure, as well as the scene transition calls across all the scenes. I used online reference to love2d and lua in coming up with this format, and used the help of genAI copilot in helping me with the code format and debaugging.

    DESCRIPTION: - What functionality does your code sample abstract away?
    The process of changing between menu items in the sub-menus of the game, so that no files need to be called explicitly. Instead of directly loading a file of the sub-menu, and creating a format to account for user interaction -- such as mouse clicks or pressing keys, but also accounting for love2d core operations -- updating, drawing and loading. This way I can have a structured control over all transitions abstracted to the form of "scenes" instead of specific file paths.
    - How does it mechanical achieve abstraction?
    It is done by the assignment of a new "current" scene to the scene provided by a current "current" scene. The program starts with an initial scene of logo.lua, proceeds to the login scenes and to the main scene of the game, and then the user can transition between the scenes by interacting with the user interface. The role of manager.lua is to take in the directions from scenes on what scene comes next -- whether automatically (like logo--login transition) or on demand (button press in menu) and call for the scene information to be loaded. This includes retriving the file data through loadScene(), ensuring the existance of a file, and then facilitating all user-interaction funcitons (key press, mouse press, text input) and love2d core functionalities -- drawing, updating and loading.
    - What purpose does this abstraction serve in your larger program?
    It simplifies the code significantly. The abstraction takes what all the scenes have in common -- the need of handling user input and being responsive to love2d operations. It adds the conceptual structure to the way the scenes operate and provides a clean functionality that requires a potential developer to only concern themselves with single-scene functionality but no need to be call each of the scene functionalities explicitly in the caller snippet.

        function stats_init.data_setup()
            -- if no game existing -- start new
            if not is_game_existing() then
                os.execute("mkdir -p " .. dir)
                stats_init.data = {}
                stats_init.create_new_player()
            -- if game existing -- load data from file to data table
            else
                local file = io.open(file_path, "r")
                local contents = file:read("*a") -- read the contents of the file
                file:close()

                if contents and contents ~= "" then
                    stats_init.data = lume.deserialize(contents)
                else
                    stats_init.data = {}
                    stats_init.create_new_player()

  7. Design: Evaluation

    8. WHERE: I recorded notes from our interaction presented below


    MY CONTRIBUTION: Performing the game presentation and interview with the user


    DESCRIPTION
    - What was the design of your user study to evaluate your artifact?
    I presented the application to a fellow Grinnell student and asked them to play with it for 15 minutes - similarly to what the initial needfinding study looked like. I presented them with the application on the desktop and asked to interact with it as if they were playing any game, and informed them that the general purpose of the game is to pass the semester as a Grinnellian student.
    - What were the results of this user study?
    I observed the user while they were interacting with the application and then asked for their impression of the experience.
    Observation + Feedback:
    - The user really enjoyed selecting the character design
    - At the start they went through the few menu options and explored their possibilities of different activites. After a minute spent browsing around the different options and the bar on the main menu, they started interacting with the game by pressing the age-up button (accidentally).
    - The user selected activities within the club and relationships before they ran out of the actions for the round, which they were surprised by. It appeared as they did not realize that there is only a set number of actions they can perform at a time, which is a feature distinct from bitlife and that may be the source of confusion.
    - After the display of pop-up window showing poor academic standing, they swearved towards having the character spend their time studying, which intended to improve their deteriorating academic performance. This continued for the next few age-up rounds and the user started to seem surprised or confused with the reason for such situation. They "were doing all they can" to improve their academic standing but felt as if the game was not responsive to their actions.
    - By the week 8 they asked whether there is a "drop out" option in the game. Then, they proceeded to stop trying to improve their academic standing and instead explored the non-academic forms of activites. They enjoyed the humor of the options and the descriptions within them.
    - They found it disappointing that the information displayed in response to their actions (the pop up window) is the same between the game rounds.
    - At week 10 they engaged in jiu-jitsu and were audibly surprised by the improved performance. They continued on pressing it until the final week of the game.
    What were the main points of future development that you took away from this activity?
    - Clearer instructions: It would be benefitial to make it clear what is the functionality of the age-up button, as well as that there are limited actions. We could implement a simple tutorial information displayed at the first run of the game.
    - Diverse information in pop-up windows in response to actions. We could use a source file with different potential results of the actions, so that the game is more entertaining and the user can keep performing the same action with new results.
    - Adding a drop-out option: A consideration - maybe it should be possible, or maybe the suffering of playing towards the end is our goal?
    - Weighing statistics: It is a fun feature of the game to discover that jiu-jitsu is the ultimate reason for a game win, but the statistics should be better responsive to the actions of the user so that they don't feel discouraged.
