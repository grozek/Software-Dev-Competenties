DESCRIPTION: - What functionality does your code sample abstract away? The process of changing between menu items in the sub-menus of the game, so that no files need to be called explicitly. Instead of directly loading a file of the sub-menu, and creating a format to account for user interaction -- such as mouse clicks or pressing keys, but also accounting for love2d core operations -- updating, drawing and loading. This way I can have a structured control over all transitions abstracted to the form of "scenes" instead of specific file paths. - How does it mechanical achieve abstraction? It is done by the assignment of a new "current" scene to the scene provided by a current "current" scene. The program starts with an initial scene of logo.lua, proceeds to the login scenes and to the main scene of the game, and then the user can transition between the scenes by interacting with the user interface. The role of manager.lua is to take in the directions from scenes on what scene comes next -- whether automatically (like logo--login transition) or on demand (button press in menu) and call for the scene information to be loaded. This includes retriving the file data through loadScene(), ensuring the existance of a file, and then facilitating all user-interaction funcitons (key press, mouse press, text input) and love2d core functionalities -- drawing, updating and loading. - What purpose does this abstraction serve in your larger program? It simplifies the code significantly. The abstraction takes what all the scenes have in common -- the need of handling user input and being responsive to love2d operations. It adds the conceptual structure to the way the scenes operate and provides a clean functionality that requires a potential developer to only concern themselves with single-scene functionality but no need to be call each of the scene functionalities explicitly in the caller snippet.
